\section{Poitioning of the Work and Interpreting the Workflow as Geodata Updating}

The results of this thesis should be understood as an evaluation of an update-oriented geospatial workflow rather than as a conventional building extraction benchmark. The aim was not to replace an existing building footprint dataset, but to test whether already available polygons can be assessed and selectively improved with recent vision and multimodal AI methods. This follows the logic of geodatabase updating, where the focus lies on identifying and modifying features that require correction instead of reconstructing an entire dataset from scratch \cite{Qin_changedetection}.

This perspective also distinguishes the workflow from most building footprint extraction approaches. Methods such as patch-based CNNs, fully convolutional networks, SegNet, U-Net and related segmentation architectures mainly aim to derive building footprints directly from high-resolution imagery \cite{Luo_overviewbuioldingfootprint, rastogi2022automatic}. In contrast, this work starts from existing polygons and investigates whether a multimodal large language model can assess polygon-image discrepancies, while SAM is used to generate refined geometry candidates \cite{bommasani2022opportunitiesrisksfoundationmodels, wu2023multimodallargelanguagemodels, Yin_2024MLLM, kirillov2023segment}.

This is particularly relevant for heterogeneous and partly informal settlement structures in the Global South, where available building datasets can differ in completeness and geometric consistency \cite{sirko2021continentalscalebuildingdetectionhigh, CHAMBERLAIN2024102104}. The following discussion therefore focuses on where this proof-of-concept workflow produced meaningful improvements, where the MLLM or SAM introduced new errors, and what this implies for future human-in-the-loop geospatial data maintenance workflows.

\section{What Types of Errors Were Corrected Successfully?}

Discuss the error composition. This should focus on the strong reduction of shifted polygons and the remaining difficulties with shape mismatch, missing parts, vegetation, and overcorrection.

Main points:
\begin{itemize}
\item shifted polygons were the most successfully corrected error type;
\item shape mismatch was reduced but remained relevant, especially in more complex areas;
\item vegetation and roof occlusion increased the risk of new post-processing errors;
\item this shows that the workflow is better at positional correction than at fully reliable semantic reconstruction.
\end{itemize}

\section{Role of the MLLM in Quality Assessment}

Discuss the MLLM separately from SAM. This is important because your workflow has two model roles: semantic assessment and geometric segmentation.

Main points:
\begin{itemize}
\item the MLLM was useful for routing buildings into full-house, partial-house, and no-visible cases;
\item it helped structure the workflow without a reference dataset;
\item however, semantic error descriptions were not fully reliable;
\item predefined categories were more evaluable than free-text descriptions;
\item exact category agreement was low, so the MLLM should be seen as a weak semantic assistant rather than a reliable standalone evaluator.
\end{itemize}

\section{Role of SAM and Post-processing in Geometry Refinement}

Discuss SAM and the deterministic post-processing as one connected refinement system. The raw SAM output was more detailed, while post-processing made it more usable as vector data.

Main points:
\begin{itemize}
\item SAM introduced image-derived boundary detail;
\item raw SAM geometries were too complex/noisy for direct GIS use;
\item post-processing reduced complexity and produced cleaner polygons;
\item full-house cases were more stable than partial-house cases;
\item partial-house cases are more difficult because the complete roof first has to be rediscovered.
\end{itemize}

\section{Transferability Across Global South Study Areas}

This should be one of the most important discussion sections. Compare Liberia, Mexico, Mozambique, Nepal, and Niger not only by performance, but by why performance differed.

Main points:
\begin{itemize}
\item Niger and Mozambique had the strongest final results and more stable area preservation;
\item Nepal had the weakest final quality, strongest area changes, and highest geometry complexity increase;
\item Mexico and Liberia were intermediate but differed in vegetation context and area changes;
\item transferability is possible, but strongly depends on settlement morphology, image quality, vegetation, and initial polygon quality.
\end{itemize}

\section{Comparison with the Switzerland Reference Test}

Use the Switzerland test as a contrast case. This is useful because it shows that the workflow is not automatically better in high-quality data contexts. In the Switzerland test, the regularized output performed slightly worse than the Microsoft source polygons against TLM at the strictest IoU threshold.

Main points:
\begin{itemize}
\item in high-quality, well-aligned data contexts, correction can remove relevant geometric detail;
\item the workflow is more useful where existing data contain visible errors;
\item this supports the interpretation that the method should be applied selectively, not globally to all polygons.
\end{itemize}

\section{Limitations of the Proof-of-Concept}

This should be honest and quite concrete.

Main points:
\begin{itemize}
\item manual evaluation is subjective and limited to a subset;
\item not all buildings passed through all pipeline stages;
\item no authoritative reference data existed for the Global South study areas;
\item the MLLM sometimes overpredicts semantic error categories;
\item free-text descriptions were often repetitive or vague;
\item SAM outputs depend strongly on prompts and image context;
\item post-processing can introduce new errors, especially in vegetation-rich or complex scenes.
\end{itemize}

\section{Implications for Human-in-the-loop Geospatial QA}

This is the “so what?” section. It should explain what your results mean for practical GIS workflows.

Main points:
\begin{itemize}
\item the workflow is not ready as a fully automatic replacement for human mapping;
\item it could be useful as a candidate-generation or triage tool;
\item the system can identify problematic polygons and suggest corrected geometries;
\item human validation remains necessary, especially for complex cases;
\item this fits semi-automatic data maintenance workflows.
\end{itemize}

\section{Future Work}

End with concrete next steps.

Main points:
\begin{itemize}
\item test larger samples and additional regions;
\item include authoritative reference data where available;
\item improve prompt engineering and model routing;
\item fine-tune or adapt the visual model for informal settlements;
\item improve vegetation and occlusion handling;
\item integrate uncertainty scores before accepting automatic corrections;
\item compare against standard segmentation baselines such as U-Net or other building extraction models.
\end{itemize}
